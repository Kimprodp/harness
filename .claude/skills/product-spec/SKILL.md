# Product Spec — 솔루션 구체화

## 역할

`idea-validation`으로 **"이 문제가 실재한다"**는 걸 검증한 후, **"그래서 무엇을 만들 것인가"** 를 구체화한다.

idea-validation이 **허점 발굴**이라면, 이 스킬은 **솔루션 설계 인터뷰**다.

---

## 핵심 원칙

### 1. 기능이 아니라 결과로 시작하라
"알림 기능" ❌ → "사용자가 알림을 받은 후 3일 내 이탈률이 낮아진다" ✅

### 2. 차별점은 "우리만이"로 시작해야 한다
"빠르다" ❌ (모두가 말함)
"우리만이 <특정 세그먼트> 에게 <특정 이점>을 제공한다" ✅

### 3. 성공 지표는 반드시 실패 신호와 쌍으로
성공만 있고 실패 기준 없는 계획은 **피드백을 받을 수 없다.**
"DAU 1000명 달성" 성공 → "3개월 연속 DAU < 200" 실패 신호

### 4. "Premises"를 합의한다
이후 모든 결정의 전제가 되는 가정을 명시적으로 적는다.
전제가 틀리면 모든 판단이 흔들린다.

---

## 인터뷰 섹션

### 섹션 1. 핵심 가치 제안 (Value Proposition)

- "한 문장으로: 누구에게 / 어떤 문제를 / 어떻게 해결하는가?"
- "이 문장을 타겟 사용자에게 그대로 보여주면 '그게 나다'라고 할 수 있는가?"
- 모호하면 → 구체적 예시를 강요한다

### 섹션 2. 핵심 기능 (Core Features)

- "성공에 필수적인 기능 3~5개를 말하라."
- 각 기능에 대해:
  - **결과**: 이 기능이 사용자에게 가져오는 결과는?
  - **대안 처리**: 이 기능이 없다면 사용자는 어떻게 하나?
  - **첫 사용자의 행동**: 이 기능을 처음 쓸 때 사용자가 무엇을 하는가?
- **5개 초과 시 push**: "MVP에 4번째, 5번째가 정말 필수인가?"

### 섹션 3. 차별점 (Differentiation)

- "기존 대안(경쟁 또는 엉성한 해결책) 대비 이것이 이기는 지점은?"
- "우리만이 ___" 형태로 한 문장 작성
- 모호하면 push: "'빠르다/편하다' 말고 왜 이것만이 이걸 할 수 있는지 구조적 이유는?"

### 섹션 4. 성공 지표 + 실패 신호 (Success + Failure)

- **정량**: 숫자로 측정 가능한 지표 2~3개 (DAU, 전환율, 결제 전환, 재방문)
- **정성**: 사용자 피드백 기준 (NPS, 특정 반응)
- **실패 신호**: 이 조건이면 피벗/중단 — 구체적 임계값

### 섹션 5. 범위 경계 (Scope Boundaries)

- **명시적 포함**: MVP에 들어갈 것
- **명시적 제외**: 빼기로 한 것 + **왜 뺐는지**
- **나중에 (추후 고려)**: 지금은 아니지만 가능성 있는 것

### 섹션 6. 접근 방식 비교 (Approaches)

idea-validation에서 접근 A, B를 가져온다면 여기서 세부 비교:

| | 접근 A | 접근 B | 접근 C |
|---|---|---|---|
| 핵심 가치 | | | |
| 공수 | | | |
| 위험 | | | |
| 차별화 | | | |
| 추천 여부 | | | |

**권장 접근**을 선택하고 근거를 명시한다.

### 섹션 7. 전제(Premises) 합의

> **이 전제들 중 하나라도 틀리면 이 계획은 무효다.**

3~5개의 핵심 가정을 명시:
- 전제 1: <예: "타겟 사용자 월 10만원 이상 지불 의사 있음">
- 전제 2: <예: "현재 수동 방식이 월 20시간 이상 소요됨">
- 전제 3:

각 전제마다 **어떻게 검증할 것인지**도 적는다.

## 안티패턴

❌ "여러 방향이 있을 수 있습니다"
❌ "둘 다 장단점이 있네요"
❌ 명확한 입장 없이 나열만 하기
❌ "나중에 결정" 떠넘기기

✅ 입장을 취한다: "접근 A를 추천한다. 이유는 ___"
✅ 선택의 기준을 명시: "만약 ___이 바뀌면 B로 갈 것이다"
✅ 전제를 흔들어본다: "이 전제가 틀리면 어떻게 되나?"

---

## 실행 절차

1. `.draft/idea-validation-*.md` 읽기 (이전 세션 산출물 참조)
2. 섹션 1~7 순차 인터뷰
3. 각 섹션 완료 시 요약 + 사용자 확인
4. 최종 산출물을 `.draft/product-spec-<YYYY-MM-DD>.md`에 저장
5. `plan.md` 작성 시:
   - 섹션 1 → plan.md §1, §3
   - 섹션 2 → plan.md §3 (주요 기능)
   - 섹션 3 → plan.md §3 (차별점)
   - 섹션 4 → plan.md §5 (성공 지표)
   - 섹션 5 → plan.md §4 (MVP 정의) + feature별 prd.md 참조
   - 섹션 6 → `context.md` ADR로 기록
   - 섹션 7 → `context.md` 제약 사항 / 리스크 섹션
