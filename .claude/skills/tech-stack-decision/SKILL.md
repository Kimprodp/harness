# Tech Stack Decision — 프로덕트 레벨 기술 결정

> **원본**: [gstack/plan-eng-review](https://github.com/garrytan/gstack) (MIT) — 프로덕트 레벨 축약본 (Step 0 + Architecture만)
> **관련**: 기능 단위 상세 설계는 `tech-spec` 스킬

## 역할

프로덕트 전체의 **기술적 뼈대**를 결정한다:
- 언어 / 프레임워크 / 데이터베이스 / 배포 방식
- 주요 아키텍처 패턴 (모놀리식 / 마이크로서비스 / 서버리스 등)
- 외부 의존성 (결제, 인증, 알림 등)

**핵심**: 세부 구현이 아니라 **"앞으로 3~6개월간의 기술 방향"** 을 잡는 것.

---

## 핵심 원칙

### 1. 지루한 기술이 먼저 (Boring Tech First)
- **Layer 1** (내장): 언어 표준 라이브러리 / 프레임워크 기본
- **Layer 2** (검증된): 널리 쓰이는 성숙한 라이브러리
- **Layer 3** (일반 원리): 직접 만드는 것이 더 나을 때만
- **EUREKA** (역발상): 산업 표준과 어긋나는 특별한 통찰이 있을 때만

**기본값은 Layer 1-2**. Layer 3, EUREKA 선택 시 명시적 근거 필요.

### 2. 완결성 편향 (Completeness over Shortcuts)
AI가 구현 시간을 압축하므로, 단축 대신 **완전한 해결책**을 선호한다:
- 풀 에러 핸들링 > 해피 패스만
- 테스트 포함 > 테스트 나중에
- 관찰성 포함 > 프로덕션에서 추가

### 3. 되돌릴 수 있는 결정 우선
불확실할 때는 **되돌리기 쉬운 선택**을 한다:
- 예: DB는 바꾸기 어렵다 → PostgreSQL 기본 (검증된 표준)
- 예: 프레임워크는 바꾸기 어렵다 → 팀 경험 있는 것 우선

## Step 0: 범위 도전 (Scope Challenge)

**리뷰 시작 전**, 먼저 기술 범위가 적절한지 검증한다.

### 체크리스트
- [ ] **기존 코드 감사**: 이 프로젝트가 **완전히 새로운가**? 기존 자산 재활용 가능한 부분은?
- [ ] **최소 변경 식별**: 핵심 목표와 nice-to-have를 구분했는가?
- [ ] **복잡도 임계값**: 새 서비스/인프라 컴포넌트가 **3개 이상**인가? 그렇다면 축소 제안
- [ ] **TODOS 교차 참조**: 이미 논의/결정된 기술 결정이 있는가?
- [ ] **배포 체계 확인**: 새 아티팩트 (CLI, 패키지, 컨테이너) 도입 시 CI/CD 계획 있는가?

### 복잡도 높으면 사용자에게

```
현재 계획이 <이유>로 복잡도가 높다.
다음 중 선택:
A. 그대로 진행 (복잡도 감수 이유 명시 필요)
B. 범위 축소 제안 (어떤 것을 뺄지 구체 제시)
C. 단계적 도입 (Phase 1은 최소, 이후 확장)
```

## 4가지 핵심 결정 영역

### 영역 1. 언어 / 프레임워크 / 런타임

각 영역(프론트 / 백 / 모바일 등)별로:

| 질문 | 답변 |
|---|---|
| 선택 | |
| Layer (1/2/3/EUREKA) | |
| 선택 근거 | |
| 고려한 대안 | |
| 대안 거절 이유 | |
| 팀 경험 수준 | |

### 영역 2. 데이터 저장소

| 질문 | 답변 |
|---|---|
| 주 DB | <PostgreSQL / MySQL / MongoDB / Firestore> |
| 선택 근거 | |
| 캐시 필요? | Redis / Memcached / 없음 |
| 파일/객체 저장? | S3 / GCS / 로컬 |
| 검색 필요? | Elasticsearch / DB 내장 / 없음 |

### 영역 3. 아키텍처 패턴

| 질문 | 답변 |
|---|---|
| 모놀리식 vs 마이크로서비스 | |
| 동기 vs 비동기 | |
| 서버리스 활용? | |
| API 스타일 | REST / GraphQL / gRPC |
| 인증 방식 | JWT / Session / OAuth 2.0 |

### 영역 4. 배포 / 운영

| 질문 | 답변 |
|---|---|
| 호스팅 | AWS / GCP / Azure / Vercel / 자체 |
| 컨테이너화? | Docker / 없음 |
| CI/CD | GitHub Actions / GitLab CI / ... |
| 관찰성 (Observability) | Datadog / Grafana / CloudWatch / 기본 |
| 에러 추적 | Sentry / Rollbar / 없음 |

## 외부 의존성 체크리스트

새 외부 서비스 추가 시 각각:

- [ ] **비용 예측**: 월 사용량 × 단가
- [ ] **대안**: 대체 가능한 서비스 / 자체 구현 비용
- [ ] **락인**: 이 서비스에 갇히는 정도 (높으면 경계)
- [ ] **장애 시 영향**: 이 서비스 다운되면 우리 서비스는?
- [ ] **규제/컴플라이언스**: GDPR, 개인정보보호법 영향
- [ ] **SLA**: 필요한 가용성 vs 제공되는 것

## 기술 리스크 + 대응책

| 리스크 | 영향도 | 대응책 |
|---|---|---|
| <예: 팀 Kotlin 경험 부족> | 중 | 첫 2주 학습, 시니어 리뷰 필수 |
| <예: PostgreSQL 운영 경험 부족> | 상 | 관리형 서비스 사용 (RDS/Cloud SQL) |

리스크만 나열 X — **대응책 필수**.

## 실행 절차

1. 이전 산출물 읽기:
   - `plan.md` §1~5 (무엇을 만드는가)
   - `.draft/scope-review-*.md` (범위)
   - `.draft/product-spec-*.md` (기능 목록)
2. Step 0 Scope Challenge 수행
3. 복잡도 높으면 사용자에게 축소 제안
4. 4개 영역 순차 결정 (사용자와 대화)
5. 외부 의존성 감사
6. 기술 리스크 + 대응책 수립
7. 산출물을 `.draft/tech-stack-decision-<YYYY-MM-DD>.md`에 저장
8. `plan.md` §6 기술 스택 + `context.md` ADR 에 반영
