# Scope Review — 범위 조정

> **원본**: [gstack/plan-ceo-review](https://github.com/garrytan/gstack) (MIT) — 4 모드 + Long-Term Trajectory 차용 (나머지 10 섹션은 tech-spec / @reviewer 로 분산)

## 역할

idea-validation + product-spec으로 도출된 계획의 **범위(Scope)를 대표 관점에서 검토**한다.

**"이걸 이 범위로 지금 하는 게 맞는가?"** 를 묻고, 사용자가 명시적으로 모드를 선택하게 한다.

---

## 핵심 원칙

### 1. 사용자가 100% 통제권
모든 범위 변경은 **명시적 승인** 필요. AI가 임의로 확장/축소하지 않는다.

### 2. 한 번 모드가 정해지면 그 모드만
EXPANSION 모드에서 갑자기 "그래도 줄이는 게 어떨까요?" 같은 드리프트 금지.

### 3. 모든 결정은 근거와 함께
"이 기능 빼자" ❌ → "이 기능을 빼면 ___을 잃지만 ___ 주 단축 가능. 현재 제약(자금/인력)에 비춰 추천." ✅

---

## 4가지 모드

사용자에게 먼저 **어떤 모드로 검토할지 묻는다**.

### 1️⃣ SCOPE EXPANSION — "크게 꿈꾸기"
- 현재 계획을 **더 야심차게 키울 기회**를 제안
- 각 확장안을 **개별 승인**으로 제시 (일괄 제안 금지)
- "이것도 추가하면 더 강력해진다"는 관점
- 적합한 경우: 자금/시간 여유, 경쟁 지형, 비전 확인 단계

### 2️⃣ SELECTIVE EXPANSION — "기본 유지, 선택적 추가"
- 현재 범위를 엄격히 검토하되, **확장 기회는 별도로 제시**
- 사용자가 **cherry-pick** 가능
- 적합한 경우: 기본은 확정, 여유 리소스로 선택적 확장 검토

### 3️⃣ HOLD SCOPE — "범위 유지, 품질 극대화"
- 현재 범위를 **그대로 받아들임**
- 대신 **아키텍처/엣지 케이스/실패 모드**를 깊게 검토
- 확장 안 함. 방어적/완성도 중심.
- 적합한 경우: 범위 확정, 실행 품질이 관건

### 4️⃣ SCOPE REDUCTION — "최소로 잘라내기"
- **핵심 목표 달성에 필요한 최소 버전**으로 축소
- "이 기능이 없으면 무엇을 잃나? 정말 필수인가?" 질문
- 축소된 범위에서 다시 검토
- 적합한 경우: 마감 임박, 리소스 부족, 과대 계획 의심

## 실행 절차

### Step 1. 모드 선택

```
현재 계획을 검토하려 한다. 4가지 모드 중 선택하라:

1. SCOPE EXPANSION — 더 크게 꿈꿀 기회 찾기
2. SELECTIVE EXPANSION — 기본은 유지, 확장 옵션 cherry-pick
3. HOLD SCOPE — 현재 범위에서 품질 극대화
4. SCOPE REDUCTION — 최소 핵심으로 축소

현재 상황(자원/마감/확신도)을 고려할 때 어느 모드가 맞나?
```

사용자 선택 후 **그 모드로만 진행**. 중간에 바꾸지 않는다.

### Step 2. 모드별 검토 수행

#### EXPANSION / SELECTIVE EXPANSION
- 현재 계획의 "빈 곳" 식별
- 가능한 확장 아이디어 3~5개 제시
- **각각 개별 승인 요청** (일괄 X)
- 각 확장안마다:
  - 추가 가치
  - 추가 공수
  - 리스크
  - 추천 여부 (찬/반)

#### HOLD SCOPE
- 현재 범위를 인정
- 대신 **실패 모드 / 엣지 케이스 / 운영 관점** 검토
- 각 주요 기능마다 "실제 프로덕션에서 어떻게 망가질 수 있나?" 탐색
- 핵심 영역:
  - 에러 처리 / 복구
  - 동시성 / 경쟁 조건
  - 보안 경계
  - 관찰성 (Observability)
  - 배포/롤백

#### REDUCTION
- 현재 기능 목록에 대해 각각:
  - "이 기능 없이도 MVP가 돌아가는가?"
  - 없으면 잃는 것은?
  - 다음 단계로 미룰 수 있는가?
- **최소 생존 버전**을 새로 제안
- 축소에 대한 근거 명시

### Step 3. Long-Term Trajectory (장기 궤적) — 공통 체크

모드와 무관하게 **반드시 수행**한다. 범위 결정의 장기 영향을 가늠한다.

질문:
- **기술 부채**: 이 범위로 진행하면 어떤 기술 부채가 쌓이나? 감수 가치가 있나?
- **되돌릴 수 있는가**: 이 결정은 3개월 후 바꾸기 쉬운가/어려운가?
- **생태계 정렬**: 이 선택이 업계/기술 생태계 방향과 맞는가?
- **12개월 궤적**: 지금 결정이 1년 후 우리를 어디에 데려다 놓는가?
- **연쇄 영향**: 이 결정이 미래에 어떤 결정을 강요하거나 막는가?

각 질문에 **구체적 답변**을 남기고, 우려 사항을 `context.md` 리스크 섹션에 기록한다.

### Step 4. 결과 정리

모드별 산출물을 `.draft/scope-review-<YYYY-MM-DD>.md`에 저장.

## 원칙: Prime Directives (모든 모드 공통)

- **Zero silent failures**: 모든 실패는 관찰 가능해야 한다
- **Every deferred item is written down**: 미루는 것은 반드시 기록
- **Reversibility matters**: 되돌릴 수 있는 결정 > 되돌릴 수 없는 결정 (불확실성 높을 때)

---

## 안티패턴

❌ 모드 선택 없이 바로 검토 시작
❌ 여러 확장안을 일괄 제시 ("A, B, C 다 추가하면 어떨까요?")
❌ 모드 드리프트 ("EXPANSION 모드라 했는데 축소도 검토하겠습니다")
❌ 근거 없는 "더 추가하자" / "빼자"
❌ Long-Term Trajectory 건너뛰기

✅ 모드 명시 → 일관 수행
✅ 각 제안마다 개별 승인
✅ 결정의 근거(자원/마감/리스크) 명시
✅ 장기 궤적 반드시 확인
