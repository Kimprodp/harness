# 할 일 체크리스트 (Tasks)

> **역할**: 진행 중인 작업의 **자연어 인덱스**
> **갱신 주기**: 매 단계 완료 시
> **Last Updated**: <YYYY-MM-DD>

---

## 📐 문서 위치 — 진행 인덱스 레이어

| 문서 | 안정성 | 역할 |
|---|---|---|
| plan.md | ⭐⭐⭐ 안정 비전 | 왜 / 무엇 |
| context.md | ⭐⭐ 결정 이유 | 어떻게 / 왜 그렇게 |
| **tasks.md** | ⭐ **변동 ↑** | **지금 무엇 (자연어 인덱스)** |
| features/\<이름\>/tech-spec.md | 변동 | 코드 디테일 (AC, 시그니처) |
| 커밋 + 코드 | 매우 변동 | 작업 디테일 SoT |

### 두 층 구조

- **tasks.md** = "사람이 읽는" 자연어 인덱스 — 비개발자도 "지금 어느 단계인지" 파악 가능
- **features/\<이름\>/tech-spec.md** = "세션이 작업하는" 코드 수준 디테일 (AC, 함수 시그니처, 엣지 케이스)
- `/task` 커맨드는 둘 다 자동 로드 — 사용자는 tasks.md 만 봐도 OK, 세션은 tech-spec 까지 다 읽음

### ✅ DO — tasks.md 에 적기 좋은 것

- 진행 중 작업의 **현재 단계 + 무엇 하는 중인지** (자연어)
- 완료 기능의 **단계 인덱스 + 커밋 해시** (git history 와 1:1 매핑)
- 한 줄 요약 (성과 + 미달 모두 명시 — 성공만 적는 함정 회피)

### ❌ DON'T

- ❌ T1/T2/T3 같은 task 접두사 → 자연어 단계명 (`mock 데이터 정비`, `메트릭 측정 모듈 도입`)
- ❌ 코드 식별자 직접 노출 (`validate_round_n_test`) → tech-spec.md 안에서만
- ❌ 완료 항목에 AC 그대로 남기기 → 단계 + 커밋 해시 + 코멘트로 압축
- ❌ 측정값/토큰 수/시간 raw 디테일 → ADR (핵심 효과만), 측정 로그 (raw)
- ❌ 운영 메타 ("본 묶음 PR 정책", "이번 주 목표") → 헤더 요약 3줄로 충분

- **다음 후보**: <간단>
- **블로커**: <간단> 또는 "없음"

## 📏 Effort 척도

| 표기 | 의미 | 예상 시간 |
|---|---|---|
| **S** | Small | 2시간 이내 |
| **M** | Medium | 반나절~1일 |
| **L** | Large | 2~3일 |
| **XL** | Extra Large | 1주 이상 (쪼개는 것 검토) |

## 📝 작업 항목 포맷

### 진행 중인 기능 — 자연어 인덱스

```markdown
### 기능: <기능명> (진행 중)

> 단계: 2/4 — <자연어 단계명>
> 무엇: <비개발자도 이해할 수 있는 한 줄 설명>
> 다음: <다음 단계 자연어>

PRD: features/<이름>/prd.md
설계: features/<이름>/tech-spec.md  ← 코드 수준 AC 는 여기

#### 단계
- [ ] 1. <단계명>
- [x] 2. <단계명> (`커밋해시`) ← 완료된 단계는 커밋 해시 명시
- [ ] 3. <단계명>
```

> 각 단계의 코드 수준 AC, 함수 시그니처, 엣지 케이스는 `features/<이름>/tech-spec.md` 에.
> tasks.md 는 "지금 어느 단계인지" 만 자연어로.

### 작은 작업 (features/ 디렉토리 안 만드는 경우)

```
- [ ] **<작업 제목>** [Effort]
  - AC: <완료 기준 1 — 측정 가능하게>
  - AC: <완료 기준 2>
  - 의존: <선행 작업명> (선택)
```

> **AC (Acceptance Criteria)** = 완료 기준. "언제 이 작업이 끝났다고 할 수 있는가?"
> 모호한 AC는 피합니다. "로그인이 된다" ❌ → "이메일+비밀번호 입력 후 대시보드로 리다이렉트된다" ✅

### 단계 단위 커밋

- 한 단계 완료 → 세션이 자동으로 커밋 메시지 제안 → 사용자 승인 → 커밋
- 푸시는 사용자 수동
- 완료 항목에 각 단계 끝의 `(커밋해시)` 표기 → 6개월 후에도 git log 와 1:1 매핑

## ✅ 완료된 기능

> **기능 하위 모든 단계 완료 시**: AI가 감지하고 아카이브 제안 → 승인 시 아래 형식으로 **압축 이동**.
> AC 그대로 옮기지 않고, 단계 + 커밋 해시 + 코멘트만 남깁니다 (인덱스 역할 회복).

### 표준 완료 패턴

```markdown
### 기능: <기능명> (완료: YYYY-MM-DD)

> ADR: [`context.md`](context.md) `[feature: <기능명>]` 태그
> PRD: [`features/<기능명>/prd.md`](features/<기능명>/prd.md)

<한 줄 요약 — 어떤 작업이고 결과가 무엇인지. 성과 + 미달 모두 명시>

#### 단계
1. <단계명> — <짧은 설명> (`커밋해시`)
2. <단계명> — <짧은 설명> (`커밋해시1`, `커밋해시2`)
3. ...

#### 코멘트 (변경/보류/후속)
- <중요 변경, 보류 사항. 상세는 ADR 참조>
- 미진행 후속: <항목> — 별도 묶음에서 처리.
```

### 실제 예시 (DataPilot)

```markdown
### 기능: analysis-performance (완료: 2026-05-08)

> ADR: [`context.md`](context.md) `[feature: analysis-performance]` 태그
> PRD: [`features/analysis-performance/prd.md`](features/analysis-performance/prd.md)

분석 1회의 시간/비용 최적화. **단발 비용 ~$5 → ~$1.6 (-67%)** 달성.
**시간 ≤ 3분 목표는 미달** (5:28).

#### 단계
1. mock 데이터 정비 — 시나리오 N=5 → 2~3 축소 (`bab4a45`)
2. 메트릭 측정 모듈 도입 — `MetricsCollector` + 6 에이전트 통합 (`ea52bf2`, `9b4fe4f`)
3. Prompt Caching 적용 — 6 에이전트 SystemMessage + 동적 빌드 (`2c48d49`, `63bbf2d`, `cd7c4a5`, `0aa8088`)
4. ⑤ Root Cause Reasoner 모델 Opus → Sonnet (`a4835c6`)
5. 측정 스크립트 추가 (`0f904e2`)

#### 코멘트
- 시간 ≤ 3분 미달. 병렬 분석 시도했으나 ④ Sonnet TPM 한도 초과로 보류. 재시도 조건은 ADR 참조.
- 미진행 후속: RCR 환각 회귀 단위 테스트 / manual eval 체크리스트 — 별도 묶음에서 처리.
```

### 안티패턴 (피할 것)

```markdown
❌ T1: mock 데이터 정비   ← T 접두사 금지, "1." 또는 자연어 단계명만
❌ AC: scenarios 디렉토리에서 N=5 → N=3 축소 ...   ← 완료 시 AC 그대로 남기지 말 것
❌ "본 묶음 PR 머지 준비"   ← 운영 메타 표현 금지
❌ 측정값 raw: input 12,234 tokens, output 4,521 tokens, ...   ← raw 는 측정 로그에
```
