> **역할**: 주요 결정의 이유와 참조 정보를 기록하는 **시방서**
> **갱신 주기**: 중요한 결정/변경이 생길 때마다
> **Last Updated**: <YYYY-MM-DD>

## 📐 문서 위치 — 결정 이유 레이어

| 문서 | 안정성 | 역할 |
|---|---|---|
| plan.md | ⭐⭐⭐ 안정 비전 | 왜 / 무엇 |
| **context.md** | ⭐⭐ **결정 이유** | **어떻게 / 왜 그렇게 + 현재 스냅샷** |
| tasks.md | ⭐ 변동 ↑ | 지금 무엇 |
| 커밋 + 코드 + 측정 로그 | 매우 변동 | 작업 디테일 SoT |

### ✅ DO — context.md 에 적기 좋은 것

- ADR — 의사결정의 **상황 / 선택 / 이유 / 대안과 거절 / 영향**
- 현재 구현 스냅샷 — "지금 어디까지 왔나" 의 짧은 요약
- 도메인 용어, 제약, 외부 의존성, 리스크 + 대응책

### ❌ DON'T — context.md 에 넣지 말 것

- ❌ 시도 1차/2차/3차 단계별 실패 스토리 → 결론만 한 줄로 압축. 스토리는 커밋 메시지에.
- ❌ 측정 토큰 수, 라운드 수, 시간/비용 raw 디테일 → 핵심 효과 (-17%, -67%) 만. raw 는 측정 로그.
- ❌ 코드 패턴 디테일 (함수명, 파일 경로 나열) → 영향 섹션에 1줄 정도. 디테일은 코드에.
- ❌ task 코드 (T1/T2/Phase n) → tasks.md 만의 영역.

**핵심 원칙**: "디테일이 필요하면 커밋 메시지와 코드에서 찾는 게 정상 흐름. ADR/Snapshot 은 의사결정 + 인덱스 역할."

## 의사 결정 기록 (ADR)

> 중요한 기술/설계 결정을 시간순으로 기록합니다.
> 나중에 "왜 이렇게 했지?" 싶을 때 바로 찾을 수 있어야 합니다.
>
> **프로덕트 레벨과 기능 레벨 결정 모두 여기에 통합 기록**합니다. 태그로 분류합니다.

### 태그 시스템

모든 ADR에는 `[태그]` 를 붙입니다. 기능 단위 결정도 별도 파일에 흩뿌리지 말고 여기에 통합.

| 태그 | 용도 |
|---|---|
| `[product]` | 프로덕트 전반 결정 (전략/비전 차원) |
| `[feature: <기능명>]` | 특정 기능 내부 결정 (예: `[feature: signup]`) |
| `[tech]` | 기술 스택 / 라이브러리 선택 |
| `[infra]` | 인프라 / 배포 / 운영 |
| `[security]` | 보안 / 인증 / 권한 |
| `[data]` | 데이터 모델 / 마이그레이션 |

### 템플릿
```
### [YYYY-MM-DD] [태그] 결정: <한 줄 요약>
- **상황**: 어떤 맥락에서 이 결정이 필요했나
- **선택**: 무엇을 선택했나
- **이유**: 왜 이걸 골랐나 (한두 문장)
- **대안과 거절 이유**: 무엇을 거절했고 왜 (한두 줄)
- **영향**: 트레이드오프 + 후속 작업 (큰 영향이면 tasks.md 항목으로 분리)
```

### 압축 가이드

ADR 본문에서 빼고 다른 곳에서 찾게 유도:

| 빼기 | 어디로 |
|---|---|
| 시도 1차/2차/3차 단계별 실패 스토리 | 커밋 메시지 (사람은 결론만 보면 충분) |
| 측정 raw (토큰 수, 시간 ms 단위) | 측정 로그 (`features/<>/eval/` 또는 별도 측정 디렉토리) |
| 코드 함수명/파일 경로 나열 | 코드 자체. ADR 영향에는 "API 시그니처 변경" 정도로 압축 |
| 환경 설정 변경 디테일 | 커밋 + 환경 파일 |

남기는 것: **결정의 핵심 효과 (-17%, -67% 같은 수치) 와 그 결정으로 변하는 큰 흐름**.
