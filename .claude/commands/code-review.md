# /code-review — 코드 리뷰

당신(메인 AI)은 사용자의 코드 변경사항에 대해 `@reviewer` 에이전트를 호출하여 배치 리뷰를 수행한다.
결과를 사용자에게 전달한 후 **자연 대화로 후속 조치**를 이어간다.

---

## 전제 조건

- **git 저장소에서 실행** (`.git` 존재).
- 말투 안티패턴 금지.
- 리뷰 결과는 별도 파일로 저장하지 않는다 (필요하면 사용자가 tasks.md 추가 요청).

---

## Phase 0: 선행 체크

1. **git 존재 확인**: `.git` 없으면 중단 + 이유 보고.
2. **변경사항 존재 확인**: `git status` 실행.
   - 변경 없음 → "리뷰할 변경사항이 없다" 보고 + 종료.
   - 충돌(unmerged) 상태 → 경고 후 진행 여부 묻기.
3. **@reviewer 에이전트 존재 확인**: `.claude/agents/reviewer.md` 없으면 "하니스 설치 확인 필요" 보고.

---

## Phase 1: 검토 범위 + 맥락 수집

### 1-1. 범위 결정

사용자 인자에 따라 범위를 결정한다. 자연어로 지정됨:

| 인자 예시 | 범위 |
|---|---|
| (없음) | `git diff HEAD` (working tree 전체) |
| "staged만" / "커밋 직전" | `git diff --cached` |
| "main 대비" / "main과 비교" | `git diff main...HEAD` |
| "<기능명> 관련" | `docs/features/<기능명>/` 에 언급된 파일 + git diff |
| "보안까지" / "보안 포함" | 범위 동일 + **@security 자동 연속 호출** 플래그 켜기 |
| 파일 경로 명시 | 해당 파일들만 `git diff -- <paths>` |

인자 해석이 모호하면 사용자에게 한 번 확인.

### 1-2. 변경 파일 요약

`git diff --stat <범위>` 실행하여 변경 파일 수 / 추가·삭제 라인 수를 사용자에게 간략히 보여준다.

```
변경 파일: 7개
+128 / -43 lines
```

### 1-3. 맥락 자동 추정

"무엇을 구현했는지" 를 사용자에게 매번 묻는 피로를 줄이기 위해 다음 순서로 추정한다:

1. **tasks.md 진행 중 항목**: `docs/tasks.md` 읽고 ⏳ 또는 "현재 작업 중" 표시 항목 확인.
2. **features/ 최근 수정 폴더**: `docs/features/` 하위에서 가장 최근 수정된 폴더 확인.
3. **최근 커밋 메시지**: `git log --oneline -5` 로 힌트 파악.

추정 결과를 사용자에게 한 문장으로 제시:
```
이 변경을 "<기능명>: <작업 요약>" 으로 추정합니다. 맞나요?
(다르면 수정 설명 / 맞으면 Enter)
```

사용자가 추가 맥락 제공 시 반영.

---

## Phase 2: @reviewer 배치 호출

Agent tool로 `reviewer` 서브에이전트를 호출:

```
Agent(
  subagent_type: "reviewer",
  description: "코드 변경사항 배치 리뷰",
  prompt: `
다음 범위의 변경사항을 10년차 시니어 관점으로 리뷰하라.
git diff는 Bash 도구로 직접 실행하라.

리뷰 대상 범위:
<Phase 1에서 결정된 git 명령>

컨텍스트 (무엇을 구현했는지):
<Phase 1에서 확정된 맥락 서술>

참조 문서 (Read로 로드):
- CLAUDE.md (프로젝트 원칙)
- docs/context.md (결정 기록)
- docs/features/<기능명>/prd.md (있으면)
- docs/features/<기능명>/tech-spec.md (있으면)

체크 포인트:
1. 스킬의 전체 체크리스트 (클린 코드 / 구조 / 로직 / 성능 / 동시성 / 에러 / 테스트 / 의존성 / 패키지 배치)
2. 기본 보안 체크만 (깊은 보안은 @security 영역)
3. 인증/결제/외부 API 관련 변경 감지 시 "보안 심화 감사 필요: Y" 플래그
4. Tech Proposal 섹션 (더 나은 기술 도입 제안) 적극 활용

출력은 스킬 지침의 Code Review Report 형식 준수.
`
)
```

---

## Phase 3: 결과 제시 + 요약 강조

### 3-1. 리뷰 리포트 출력
에이전트 리포트를 **원본 그대로** 사용자에게 전달.

### 3-2. 핵심 요약 강조
리포트 끝에 메인 AI가 요약을 덧붙인다:

```
─────────────────────────
📊 요약
- 🚨 Critical: N건
- ⚠️ Warning: N건
- 💡 Suggestion: N건
- 🛠 Tech Proposal: N건
- 머지 권장 여부: 🟢 / 🟡 / 🔴
─────────────────────────
```

### 3-3. 보안 플래그 조건부 출력

@reviewer 리포트에서 **"보안 심화 감사 필요: Y"** 가 반환됐으면 눈에 띄는 블록으로 알림:

```
🔒 ─────────────────────────────────────────
   보안 심화 감사 권장
   (인증 / 결제 / 외부 API 관련 변경 감지)
   
   "보안도 봐줘"라고 말씀하시면 이어서
   보안 심화 감사가 수행됩니다.
─────────────────────────────────────────
```

### 3-4. 자연 대화 안내

요약 다음에 사용자가 어떻게 진행할 수 있는지 안내:

```
이슈에 대해 자유롭게 대화하세요. 예:

• "C1, C2 지금 고쳐줘"              → 대화로 코드 수정
• "Critical 이슈 tasks.md에 추가해줘" → 할 일로 기록
• "W2는 무시해도 돼"                 → 넘어감
• "S3 더 자세히 설명해"              → 상세 설명
• "보안도 봐줘"                      → 보안 심화 감사

또는 리포트를 그대로 두고 나중에 보셔도 됩니다.
```

## Phase 4: 자연어 후속 조치

사용자 응답에 따라 메인 AI가 상황에 맞게 반응한다.

### 요청 유형별 대응

| 사용자 요청 | 메인 AI 행동 |
|---|---|
| "<이슈 ID> 고쳐줘" / 수정 요청 | 해당 파일 읽고 리포트의 수정안 기반 코드 수정. 수정 후 사용자 확인. |
| "tasks.md에 추가해줘" | `docs/tasks.md` 에 리뷰 이슈를 할 일로 추가 (아래 형식). |
| "W2 무시" / "S1 나중에" / "넘어가" | 그대로 넘어감. 추가 조치 X. |
| "<이슈 ID> 더 설명해" | 해당 이슈 맥락을 @reviewer 리포트에서 찾아 사용자에게 풀어서 설명. |
| "@security 호출" / "보안도 봐줘" | Phase 5로 진입 (@security 에이전트 호출). |
| "전부 수정해줘" | Critical/Warning 전부 순차 수정. 각 수정마다 사용자 확인 권장. |

### tasks.md 추가 형식 (요청 시)

리뷰 이슈는 **해당 기능 서브섹션 아래** `#### 리뷰 발견사항` 으로 추가한다.

예: 기능이 `signup` 이면 `## Phase 1 > ### 기능: signup > #### 리뷰 발견사항`:

```markdown
### 기능: signup
  #### 구현
    ...
  #### 테스트
    ...
  #### 리뷰 발견사항 (<YYYY-MM-DD> 리뷰)
    - [ ] **[C1] <이슈 제목>** [S/M]
      - 파일: <경로:라인>
      - AC: <리포트의 수정 방향 달성>
    - [ ] **[W1] <이슈 제목>** [S]
      - 파일: <경로:라인>
      - AC:
```

### 기능 식별이 모호할 때 (인프라/공통/다수 기능 걸침)

해당 Phase 아래 **공통 섹션**에 추가:

```markdown
## Phase 1: MVP

### 공통 / 인프라
  #### 리뷰 발견사항 (<YYYY-MM-DD>)
    - [ ] <이슈>
```

Critical 은 반드시 포함. Warning 은 사용자 선택.

## Phase 5: @security 연속 호출 (선택적)

Phase 1에서 "보안까지" 인자가 있었거나, Phase 4에서 사용자가 요청한 경우 실행.

Agent tool로 `security` 서브에이전트 호출:

```
Agent(
  subagent_type: "security",
  description: "코드 변경사항 보안 심화 감사",
  prompt: `
방금 @reviewer가 리뷰한 동일 범위를 OWASP Top 10 기준으로 심화 감사하라.
실제 악용 가능한 취약점에 집중하고, "검토 완료 (이상 없음)" 섹션을 반드시 포함하라.

감사 대상:
<Phase 1의 git 범위 동일>

컨텍스트 (무엇을 구현했는지):
<Phase 1에서 확정된 맥락>

참조 문서:
- CLAUDE.md
- docs/context.md
- docs/features/<기능명>/prd.md, tech-spec.md (있으면)

출력은 스킬 지침의 Security Audit Report 형식 준수.
사용자와 대화하지 말고 리포트만 반환.
`
)
```

결과 제시 후 Phase 4 와 동일한 자연 대화로 이어간다. High/Medium/Low 이슈에 대해 사용자가 수정·무시·기록 선택.

---

## Phase 6: 리뷰 후속 정리 (선택)

사용자가 리뷰 이슈 수정을 완료했음을 알리면(예: "전부 수정함", "리뷰 이슈 처리 완료"):

1. `docs/tasks.md` 의 `#### 리뷰 발견사항` 섹션의 해당 항목들을 체크오프 (`[x]` + 완료 스탬프).
2. **`/update-docs` 실행 권장**:

```
🔧 리뷰 이슈 모두 처리 완료.

이어서 문서도 갱신하시겠어요?
- "네" / "Y" → /update-docs 바로 실행 (ADR 추가, Implementation Snapshot 갱신)
- "나중에" → 종료

근거: 리뷰 수정에서 아키텍처 / 보안 / 기술 선택에 영향 있는 변경이 있었으면
context.md ADR 추가가 필요할 수 있습니다.
```

### 자동 연속 호출 메커니즘

사용자가 "네/Y" 응답 시:
1. `.claude/commands/update-docs.md` 를 Read 도구로 로드.
2. 그 파일의 Phase 지시를 **현재 세션 내에서 이어서 수행**한다.
3. 사용자는 재입력 불필요.

> 만약 동작이 이상하면 사용자가 직접 `/update-docs` 를 입력해도 된다.

## 핵심 원칙 (위반 금지)

1. **에이전트는 배치 호출**: @reviewer, @security 모두 사용자와 대화 시도 금지.
2. **메인 AI는 지휘자**: git diff 추출·컨텍스트 요약·사용자 대화만 메인이, 실제 분석은 에이전트.
3. **결과 저장 금지**: 리뷰 리포트를 파일로 저장하지 않는다. 필요하면 tasks.md 추가로 대체.
4. **자동 수정 금지**: 사용자 승인 없이 코드 수정하지 않는다.
5. **보안 플래그는 명시적으로**: 눈에 띄는 블록으로 표시. 그냥 한 줄로 넘기지 말 것.
6. **메뉴식 강제 금지**: 고정 선택지 A/B/C/D 제시하지 말고 자연 대화로.

---

## 실패 시나리오 대응

| 상황 | 대응 |
|---|---|
| 변경사항 없음 | "리뷰할 것 없음" 보고 후 종료 |
| @reviewer 호출 실패 | 오류 보고 + 재시도 제안 |
| 에이전트가 빈 리포트 반환 | 수동 재호출 + 컨텍스트 강화 |
| 사용자가 모호하게 답변 | 한 번만 재질문 후 사용자 의도에 맞게 진행 |
| 인증/결제 변경인데 @reviewer가 보안 플래그 안 켬 | 메인 AI가 감지 시 별도로 보안 감사 권장 안내 (과탐 OK) |
