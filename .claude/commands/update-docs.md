# /update-docs — 3문서 갱신

당신(메인 AI)은 최근 작업 내용을 바탕으로 `docs/context.md`, `docs/tasks.md`, (필요 시) `docs/plan.md` 를 갱신한다.

**언제 쓰나**:
- 긴 작업 세션 끝날 때 (컨텍스트 날아가기 전 스냅샷 기록)
- 주요 결정 내린 직후 (ADR 추가)
- 커밋/PR 직전 (문서와 코드 동기화)
- `/project-status` 에서 "문서 오래됨" 경고 받았을 때

---

## 전제 조건

- `docs/context.md`, `docs/tasks.md` 존재 필수 (없으면 `/kickoff` 안내 후 중단).
- **사용자 승인 전에 파일 수정 금지**. 제안 → 확인 → 반영 순서.
- 제안은 구체적으로 (어느 섹션 / 어떤 내용 / 왜).

---

## Phase 0: 선행 체크

1. `docs/plan.md`, `docs/context.md`, `docs/tasks.md` 존재 확인.
2. `.git` 있으면 git 활동 기반 자동 추정 활용.

---

## Phase 1: 변경사항 / 맥락 수집

다음 소스로부터 "마지막 갱신 후 무슨 일이 있었는지" 추정:

### 소스 1: git 활동

`docs/context.md` 의 **Last Updated** 값을 기준 날짜로 사용:

```bash
# Last Updated가 "2026-04-18" 이면
git log --oneline --since="2026-04-18"
git diff --stat --since="2026-04-18" HEAD
```

Last Updated가 없거나 파싱 실패 시: 최근 10커밋 (`git log --oneline -10`) 로 대체.
최근 수정 파일 목록 / 커밋 메시지에서 주요 변경 식별.

### 소스 2: 현재 대화 맥락
이 세션에서 수행한 작업 요약 (사용자 요청, 에이전트 호출, 수정한 파일 등).

### 소스 3: 기존 문서
- `docs/context.md` Implementation Snapshot 섹션 — 이전 스냅샷과 비교
- `docs/tasks.md` — 진행 중 ⏳ / 미완료 항목 중 최근 완료 가능한 것 식별

### 소스 4: 활성 기능
- `docs/features/*/tech-spec.md` 최근 수정 시점 확인

---

## Phase 2: 갱신 제안 생성

3개 영역에 대해 각각 제안:

### 2-A. context.md — Implementation Snapshot

현재 `## 현재 구현 스냅샷` 섹션을 **최신 정보로 전면 교체**.

제안 내용:
- **마지막 업데이트 시점**: <현재 YYYY-MM-DD HH:MM>
- **현재 구현 중인 것**: <대화 맥락 + git 활동 기반>
- **최근 세션에서 내린 주요 결정**: <있으면 목록>
- **최근 수정된 주요 파일**:
  - `경로/파일.ext` — <왜 수정했나>
- **현재 블로커**: <있으면>
- **다음 즉시 해야 할 일**: <tasks.md 기반 추정>
- **미완성/미커밋 변경사항**: <git status 기반>

### 2-B. context.md — ADR 추가 제안 (선택)

세션 중 **새로운 결정**이 있었으면 ADR 추가 제안:
```
### [날짜] [태그] 결정: <요약>
- 상황:
- 선택:
- 이유:
- 대안과 거절:
- 영향:
```

각 결정마다 사용자에게 **"이거 ADR로 추가할까요?"** 개별 확인.

### 2-C. tasks.md — 체크오프 + 새 작업 추가

- **완료 감지**: 진행 중 ⏳ 항목 중 대화/커밋 근거로 완료된 것 찾아 ✅ 처리 제안
- **"완료된 작업" 섹션으로 이동** 제안
- **새로 생긴 작업**: 대화 중 나온 할 일, 리뷰 이슈 등 추가 제안
- **블로커 업데이트**

### 2-D. plan.md — 보통 X, 필요 시만

`plan.md` 는 큰 방향 변경 시에만 수정. 다음 경우만 제안:
- 프로덕트 방향 변경 감지
- Phase 완료 / 다음 Phase 진입
- 새 리스크 발견
- MVP 정의 변경

일반적으론 **"plan.md 변경 제안 없음"** 으로 표시.

## Phase 3: 사용자 확인 + 반영

제안을 다음 형식으로 사용자에게 제시:

```markdown
# 📝 Update Proposal — <YYYY-MM-DD HH:MM>

## 1. context.md — Implementation Snapshot 갱신

(Before / After 비교 또는 제안 전문)

**반영?** Y / N / 수정

## 2. context.md — 새 ADR 추가 (N개)

### ADR 1: [태그] <제목>
(제안 전문)
**반영?** Y / N / 수정

### ADR 2: ...

## 3. tasks.md — 변경사항

### 완료 처리 제안 (N개)
- [x] <작업> — 근거: <커밋 / 대화 내용>

### 새 작업 추가 제안 (N개)
- [ ] <작업> [Effort] — 출처: <대화 / 리뷰 이슈>

### 블로커 변경
- <변경 사항>

**반영?** Y / N / 항목별 선택

## 4. plan.md — 변경 제안

<없음 or 구체 제안>
```

### 사용자 응답 처리

- **"Y" / "전부 반영"**: 모든 제안을 해당 파일에 실제 반영
- **"N" / "나중에"**: 반영 없이 종료
- **"1, 3만"** 또는 선택적**: 선택된 것만 반영
- **"수정 후 반영"**: 사용자 수정 요청 반영 후 반영

반영 시 반드시 **파일의 `Last Updated` 필드 갱신**.

---

## Phase 4: 반영 후 확인

반영 완료 후 사용자에게 간단 보고:

```
✅ 문서 갱신 완료

- context.md: Implementation Snapshot 갱신 / ADR N개 추가
- tasks.md: N개 완료 처리 / M개 추가
- plan.md: 변경 없음

다음: /project-status 로 갱신 상태 확인 / 계속 작업
```

---

## 핵심 원칙

1. **제안 → 승인 → 반영** 순서 엄수. 독단 수정 금지.
2. **Before/After 명확**: 무엇이 어떻게 바뀌는지 보여주기.
3. **Last Updated 갱신**: 파일 수정 시 반드시 해당 필드 최신화.
4. **ADR 태그 필수**: 새 ADR 추가 시 `[product]/[feature: X]/[tech]/...` 태그 포함.
5. **완료 이동 원칙**: tasks.md 완료 항목은 체크 + "완료된 작업" 섹션으로 이동 (삭제 금지).
6. **회귀 방지**: Last Updated 이전 내용을 덮지 말 것. 추가·수정만.

## 실패 시나리오

| 상황 | 대응 |
|---|---|
| context.md/tasks.md 없음 | `/kickoff` 먼저 필요 안내 |
| 제안할 변경 없음 | "갱신이 불필요합니다. 3문서는 최신입니다." 보고 |
| 대화 맥락 부족 | 사용자에게 "이번 세션에 주요 변경 있었나요?" 질문 |
| 충돌하는 결정 감지 | "기존 ADR과 충돌: <기존 결정>. 폐기/수정/공존 중 선택" 요청 |
| git log 접근 불가 | 대화 맥락만으로 제안 |
