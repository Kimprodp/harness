# /investigate — 디버깅 조사

당신(메인 AI)은 사용자가 겪고 있는 버그/이슈의 **근본 원인을 체계적으로 조사**한다.
증상과 원인을 분리하고, 확증 편향을 피하기 위해 **최소 3개 가설**을 세운다.

**언제 쓰나**:
- 버그가 발생했는데 원인이 바로 보이지 않을 때
- 테스트가 실패하는데 이유가 불분명할 때
- 재현은 되는데 왜 그런지 모를 때
- "왜 이게 안 되지?" 싶은 상황 전반

**언제 쓰지 말 것**:
- 원인이 이미 명확한 단순 버그 (바로 `/task` 로 수정)
- 새 기능 설계 (`/feature-plan` 영역)
- 코드 품질 검토 (`/code-review` 영역)

---

## 전제 조건

- **조사 전용 커맨드**. 이 커맨드 안에서는 **코드를 수정하지 않는다**. 실제 수정은 조사 완료 후 `/task` 또는 자연 대화로.
- **추측과 검증된 사실을 엄격히 분리**한다. 검증 안 된 가설을 결론처럼 제시 금지.
- 말투 안티패턴 금지 ("흥미롭네요", "crucial" 등).
- 가설은 **최소 3개**. 한두 개로 성급히 결론 내지 않는다.

---

## Phase 0: 선행 체크

1. **프로젝트 루트 확인**: 가능하면 루트에서 실행. 서브 디렉토리에서도 동작하나 경로 해석 주의.
2. **git 확인**: `.git` 있으면 이력 기반 검증 가능. 없으면 해당 분석 생략하고 진행.
3. **문서 참조 (선택)**: `docs/context.md` 존재 시 관련 ADR이 있는지 가볍게 확인 (조사 중 필요해지면 Read).

---

## Phase 1: 증상 수집

### 1-1. 인자 기반 파악

사용자가 인자로 이슈 설명을 줬으면(`/investigate 로그인 후 리다이렉트 안 됨`) 그것을 초기 증상으로 간주하고 상세화 질문으로 이어간다.

인자 없으면 아래 4가지를 한 번에 묻는다:

```
이슈를 조사하기 전에 알려주세요:

1. **무엇이 일어나고 있나요?** (관찰된 증상)
2. **무엇이 일어나야 하나요?** (기대 동작)
3. **재현 절차** (어떤 조작 → 어떤 에러)
4. **에러 메시지 / 로그** (있으면 그대로 붙여넣기)

모르는 항목은 "모름"이라고 답하셔도 됩니다.
```

### 1-2. 증상 정제

받은 답변을 다음 형식으로 정리하여 사용자에게 확인:

```
📋 증상 정리
- **관찰**: <실제 일어나는 일>
- **기대**: <원래 일어나야 하는 일>
- **재현**: <재현 절차, 불명확하면 "재현 불안정">
- **증거**: <에러 메시지, 스택 트레이스, 로그 등>
- **범위**: <특정 사용자 / 특정 환경 / 항상 / 간헐적>
```

사용자가 보정하면 반영 후 Phase 2로.

### 1-3. 관련 파일 자동 탐색

증상 키워드 기반으로 Grep/Glob:
- 에러 메시지 문자열 → 코드 내 해당 문자열 위치
- 함수/클래스 이름 → 정의/호출 위치
- 관련 파일 경로 목록을 "조사 후보"로 메모

---

## Phase 2: 가설 생성 (최소 3개)

### 원칙

- **3개 이상 필수**. 단일 가설은 확증 편향의 온상.
- 가설은 **서로 배타적** 이어야 함. "A 때문일 수도, A의 변종 때문일 수도"는 하나로 간주.
- 각 가설에 **반증 방법** 을 반드시 붙인다. 반증 불가능한 가설은 제외.

### 출력 형식

```
🔍 가설 후보

### 가설 1: <한 줄 서술>
- **근거**: <왜 이 가설이 가능한가, 증상과의 연결>
- **반증 방법**: <어떻게 이 가설을 부정/확인할 수 있는가>
- **우선순위**: 🟢 높음 / 🟡 중간 / 🔴 낮음
- **근거 강도**: ★★★ / ★★ / ★

### 가설 2: ...

### 가설 3: ...
```

### 가설 종류 예시 (참고)

| 카테고리 | 예시 |
|---|---|
| 최근 변경 | 방금 머지한 커밋이 깨뜨림 |
| 환경 차이 | 로컬 ok / prod 실패 → 환경변수 누락 |
| 상태 / 데이터 | 특정 DB 레코드 / 캐시 | 
| 동시성 | 레이스 컨디션, 순서 의존 |
| 의존성 | 라이브러리 버전 불일치, 업스트림 변경 |
| 로직 오류 | 조건문, 경계값, off-by-one |
| 설정 / 권한 | 퍼미션, 토큰 만료 |

---

## Phase 3: 가설 검증

### 3-1. 우선순위 순서로 검증

가설 1 (최고 우선순위) 부터 순차 검증. 검증된 가설은 **확정(🟢)** 또는 **반증(🔴)** 표시.

### 3-2. 검증 도구

- **코드 경로 추적**: Grep/Read로 실행 흐름 따라가기. 분기 조건 / 변수 상태 확인.
- **git 이력**: `git log --oneline -20 -- <파일>`, `git blame <파일> -L <라인>` 로 최근 변경자/시점 확인.
- **최근 머지 의심 시**: `git log --since="<날짜>" --oneline` 으로 가설 1에 유력.
- **환경 설정 확인**: `.env`, `config/`, `settings.json` 등 Read (민감정보는 출력 X).

### 3-3. 서브 에이전트 연동 (선택적)

검증 중 다음 상황이면 해당 에이전트 호출 제안:

| 상황 | 제안 에이전트 | 예시 |
|---|---|---|
| 인증/세션/권한 관련 증상 | @security | "로그인 후 권한 오류" |
| 성능/동시성/아키텍처 의심 | @reviewer (의견만) | "간헐적 500 에러" |

**주의**: 이들은 조사 보조용. 자동 호출 X. 사용자에게 "보안 심화 감사를 받으면 가설 N 검증에 도움됩니다. 호출할까요?" 식으로 제안.

### 3-4. 가설 전멸 시

모든 가설이 반증되면:
- Phase 2로 돌아가 **새 가설 3개** 생성 (이미 반증된 것 제외).
- 필요한 추가 증거를 사용자에게 요청 ("이 로그도 볼 수 있을까요?").
- 2회 반복 후에도 미해결이면 Phase 4로 가되 "미해결" 리포트로.

## Phase 4: Root Cause 리포트

### 출력 형식

```markdown
# 🔬 Investigation Report — <YYYY-MM-DD HH:MM>

## 증상 (요약)
<Phase 1-2에서 정리된 관찰/기대>

## Root Cause
<한 줄로 근본 원인>

- **검증된 가설**: 가설 N ("<서술>")
- **반증된 가설**: 가설 M, K
- **근거**: <검증 과정에서 확인된 증거 — 파일:라인, git commit SHA 등>

## 영향 범위
- **직접 영향**: <증상이 재현되는 조건>
- **간접 영향 추정**: <비슷한 코드 경로가 같은 원인으로 실패할 가능성>

## 재현 조건
<조건 충족 시 100% 재현되는지, 확률적인지>

## 수정 제안 (옵션)

### 옵션 A: <제목>
- **내용**: <무엇을 어떻게 바꾸는지>
- **난이도**: S / M / L
- **리스크**: <부작용 가능성>

### 옵션 B: <제목>
...

## 추천
옵션 <A/B/...> — 이유: <근거>

## 미해결 항목 (있으면)
- <Phase 3에서 가설 전멸 시 남은 의문점>
- <추가 증거가 필요한 영역>
```

### tasks.md 추가 제안

리포트 제시 후 사용자에게:

```
이 조사 결과를 tasks.md에 수정 작업으로 추가할까요?

- "네 / 추가해줘" → 해당 기능 섹션 또는 공통 섹션에 todo로 추가
- "지금 바로 고쳐줘" → /task 흐름으로 전환해 구현 시작
- "일단 기록만" → 리포트만 보고 종료
```

### tasks.md 추가 형식 (승인 시)

기능 식별 가능하면 해당 `기능: <X>` 섹션 아래, 아니면 공통 섹션에:

```markdown
  #### 버그 수정 (<YYYY-MM-DD> 조사)
    - [ ] **[B1] <Root Cause 한 줄>** [S/M/L]
      - 원인: <리포트 요약>
      - 파일: <경로:라인>
      - AC: <증상 재현 절차로 테스트 → 에러 없음>
      - 옵션: <A/B 중 선택 근거>
```

## Phase 5: 자연 대화 후속

리포트 제시 후 사용자 응답에 따라:

| 사용자 반응 | 메인 AI 행동 |
|---|---|
| "옵션 A로 고쳐줘" | 해당 파일 수정 착수 (사용자 확인 받으며) |
| "tasks.md에 추가만" | Phase 4의 형식으로 추가 |
| "영향 범위 더 봐줘" | 비슷한 코드 경로 Grep 확장 조사 |
| "재조사" / "가설 더 찾아봐" | Phase 2로 돌아가 추가 가설 생성 |
| "그냥 닫아" | 종료. 리포트는 저장하지 않음 (필요 시 tasks.md 경유) |
| "보안도 봐줘" | @security 에이전트 호출로 전환 |

---

## 핵심 원칙 (위반 금지)

1. **조사 전용**: 이 커맨드 안에서 **코드 수정 금지**. 수정은 Phase 5에서 사용자 승인 후 별도 진행.
2. **가설 최소 3개**: 단일/이중 가설로 결론 내지 않는다.
3. **증상 ≠ 원인**: "에러 메시지 X가 난다"는 증상. 원인은 그 X를 던지는 코드 경로.
4. **추측과 사실 분리**: 검증되지 않은 가설을 "원인"으로 단정 금지. "유력 가설"로 표현.
5. **반증 가능성**: 모든 가설은 반증 방법이 있어야. "그냥 느낌" 가설 배제.
6. **과도한 도구 사용 자제**: git log를 한 번에 수십 줄 뽑지 말고 범위 좁혀 여러 번.
7. **자동 에이전트 호출 금지**: @security, @reviewer 는 제안만, 실행은 사용자 승인 시.

## 실패 시나리오 대응

| 상황 | 대응 |
|---|---|
| 재현 불가 | 사용자에게 "재현 가능한 최소 절차"를 요청. 불가 시 관찰된 증거만으로 가설 도출 후 "재현 불가 — 로그 수집 필요" 리포트. |
| 증상이 모호 | Phase 1 재질문 1회. 여전히 모호하면 "범위 축소 필요" 로 리포트 종료. |
| 가설 2회 연속 전멸 | "미해결 리포트" 로 마무리 + 추가 증거 요청 명시. 무한 루프 X. |
| 보안 의심 높음 | Phase 3에서 `@security` 호출 적극 제안. |
| git 없음 | git 관련 검증 생략, 코드 경로 분석만으로 진행. |
| 너무 큰 조사 범위 | 사용자에게 "이 범위는 여러 조사로 분할 권장" 제안 후 하나만 선택. |
