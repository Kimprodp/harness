# /decision — ADR 실시간 기록

당신(메인 AI)은 사용자가 **방금 내린 결정 1건** 을 구조화된 ADR (Architecture Decision Record) 로 `docs/context.md` 에 추가한다.
맥락이 뜨거울 때 바로 기록하여 회고형 재구성의 품질 손실을 방지한다.

**`/update-docs` 와의 차이**:
- `/update-docs`: 세션 끝 회고형. Snapshot + 여러 ADR + tasks + plan 일괄.
- `/decision`: **단건 집중**. 결정 직후 5분 내 실시간 기록. 품질 높은 ADR 보장.

**언제 쓰나**:
- 대화 중 주요 결정 확정됐을 때 (PG 선택, 라이브러리 채택, 아키텍처 패턴 결정 등)
- `/task`, `/code-review`, `/ship` 진행 중 결정 포인트 마주쳤을 때
- 중요한 "거절된 대안" 이 있을 때 (Why not 을 남겨야 할 때)

---

## 전제 조건

- `docs/context.md` 존재 필수. 없으면 `/kickoff` 안내 후 중단.
- 사용자 승인 전까지 context.md 수정 금지. 드래프트 → 확인 → 반영.
- 한 번에 **ADR 1건** 만 처리. 여러 건 한꺼번에 기록은 `/update-docs` 로.

---

## Phase 0: 선행 체크

1. `docs/context.md` 존재 확인.
2. 인자로 결정 요약 받았는지 확인. 있으면 Phase 1 의 Q1 답변으로 반영 후 Q2 부터 진행.

---

## Phase 1: 대화형 질문 (8개)

사용자에게 **한 번에 모든 질문을 나열하여 제시**. 답 받으면 부족한 항목만 재질문.

```
🎯 /decision — 결정 기록

다음 8개 항목 알려주세요 (모르면 "모름"):

1. 결정 요약 (1줄)
2. 상황/배경 — 왜 선택이 필요했나
3. 고려한 선택지 — 후보 목록
4. 최종 선택
5. 주요 이유 — 3가지 이내
6. 거절된 대안과 각각의 이유
7. 영향 범위 — 비용/성능/운영/보안 등
8. 태그 — 여러 개 가능: [product] / [feature: X] / [tech] / [infra] / [security] / [data]
```

### 답변 처리 규칙

- **Q1 (요약)**: 필수. 없으면 재질문.
- **Q2~Q5**: 하나라도 "모름" 이면 1회 재질문 ("왜 선택했는지 더 설명해줄 수 있나요?"). 그래도 모호하면 **그대로 드래프트** 에 포함하되 "(보강 필요)" 플래그.
- **Q6 (거절 대안)**: 비어있어도 OK. 단, 후보가 여럿이었다면 강하게 권장.
- **Q7 (영향)**: 없으면 "추정 영향 없음" 으로 표기.
- **Q8 (태그)**: 필수. 최소 1개. 불명확하면 AI 가 자동 추정 후 사용자 확인.

---

## Phase 2: ADR 드래프트 생성

Phase 1 답변을 바탕으로 다음 형식의 ADR 드래프트 작성 (context.md 에 추가될 블록):

```markdown
### <YYYY-MM-DD> <태그들> 결정: <Q1 요약>

- **상황**: <Q2>
- **선택**: <Q4>
- **이유**:
  - <Q5 항목 1>
  - <Q5 항목 2>
  - <Q5 항목 3>
- **대안과 거절**:
  - <후보 A>: <거절 이유>
  - <후보 B>: <거절 이유>
- **영향**: <Q7>
- **관련**: <tag 가 [feature: X] 이면 해당 기능>
```

### 태그 우선순위 규칙

- 기능 특정이면 `[feature: <name>]` 우선, `[tech]` 또는 다른 태그 병기.
- 프로덕트 전반 결정이면 `[product]`.
- 여러 태그는 공백 구분. 예: `[feature: payment] [tech] [security]`

---

## Phase 3: 사용자 확인 + 반영

드래프트를 사용자에게 보여주고 승인 받기:

```
📝 ADR 드래프트

<드래프트 전문>

이대로 context.md 에 추가할까요?
- "네" / "Y" → 반영
- "수정" → 해당 섹션 수정 후 재확인
- "취소" → 반영 없이 종료
```

### 승인 시 동작

1. `docs/context.md` Read
2. ADR 섹션 위치 찾기 (`## Architecture Decision Records` 또는 유사)
3. 해당 섹션 끝에 드래프트 **append** (기존 내용 유지)
4. `docs/context.md` 의 **Last Updated** 오늘 날짜로 갱신 (freshness Hook 이 못 잡을 수 있어 명시)
5. 반영 완료 보고:

```
✅ ADR 추가 완료

- 위치: docs/context.md L<라인>
- 태그: <태그들>
- 요약: <Q1>

다음에 /project-status 실행 시 "최근 결정" 섹션에 자동 노출됩니다.
```

---

## Phase 4: 후속 대화 (선택)

결정 후 자연스러운 후속:

| 상황 | 제안 |
|---|---|
| 결정이 구현 task 를 새로 요구 | "tasks.md 에 관련 todo 추가할까요?" 1회 제안 |
| 결정이 기존 tech-spec 변경 | "`docs/features/<X>/tech-spec.md` 갱신 필요할 수 있음" 안내 |
| 결정이 리스크 수반 | "`docs/context.md` Risk 섹션에도 추가할까요?" |
| 여러 결정 동시 발생 | "다른 결정도 기록? `/decision` 다시 실행" |

단, 이 후속은 **1회만 제안**. 사용자가 "아니" 하면 바로 종료.

---

## 핵심 원칙 (위반 금지)

1. **단건 집중**: 한 번 실행 = ADR 1개. 여러 건 몰아넣지 않는다.
2. **대화형 필수**: 8개 질문 모두 거쳐야 함. AI 가 임의로 답 지어내지 않는다.
3. **사용자 승인 후 반영**: 드래프트 단계 → 승인 → 반영 순서.
4. **기존 ADR 덮어쓰기 금지**: append 만. 수정 필요하면 사용자가 직접 편집.
5. **태그 필수**: 태그 없는 ADR 생성 금지 (검색·필터링 기반).
6. **빠름 우선**: 전체 플로우 5분 이내 목표. 8개 질문 간결하게.

---

## 실패 시나리오 대응

| 상황 | 대응 |
|---|---|
| context.md 없음 | `/kickoff` 안내 후 중단 |
| context.md 에 ADR 섹션 없음 | 사용자에게 "ADR 섹션 신규 생성할까요?" 확인 후 추가 |
| Q1 요약 모호 | 1회 재질문. 그래도 모호면 "결정 정리 후 다시 호출 권장" 보고 후 중단 |
| 태그 결정 못함 | AI 가 Q2, Q5, Q7 기반 추정 → 사용자 확인 |
| 같은 결정 ADR 이미 있음 | 기존 ADR 위치 보여주고 "덮어쓰기 / 신규 추가 / 취소" 선택 |

---

## `/update-docs` 와의 분업 규칙

- **단건 결정** → `/decision` 권장
- **세션 끝 회고** (Snapshot + 여러 ADR + tasks 일괄) → `/update-docs`
- `/update-docs` 의 ADR 추가 로직은 **내부적으로 이 커맨드의 양식 재사용**. 즉 ADR 품질 기준은 동일.
