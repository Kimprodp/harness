# /task — 다음 task 진입

당신(메인 AI)은 `docs/tasks.md`의 **다음 작업에 진입**하며, 관련 맥락을 자동 로드하고 사용자와 함께 구현을 시작한다.

**언제 쓰나**:
- `/feature-plan` 완료 후 구현 시작 시
- 매 task 완료 후 다음 task로 이어갈 때
- 막혀서 다른 task로 전환할 때

---

## 전제 조건

- `docs/tasks.md` 존재 필수 (없으면 `/kickoff` + `/feature-plan` 안내).
- 관련 기능의 `docs/features/<X>/prd.md`, `tech-spec.md` 가 있으면 맥락 자동 로드.
- **맥락 로드 + 시작 확인까지만**. 구현은 자연 대화로.
- 자동 구현 금지. 사용자 승인 없이 코드 수정 X.

---

## Phase 0: 선행 체크

1. `docs/tasks.md` 존재 확인.
2. 미완료 `[ ]` 또는 진행 중 ⏳ 항목 존재 확인.
   - 모두 완료됐으면: **"현재 Phase 모든 task 완료. `/feature-start` 로 새 기능을 시작하거나 `/project-status` 로 전체 상황을 확인하세요."** 보고 후 종료.

---

## Phase 1: Task 선택

### 1-1. 인자 기반 (사용자 지정)

사용자가 인자를 줬으면 우선 매칭:

| 인자 예시 | 매칭 방법 |
|---|---|
| `/task JWT` | tasks.md 제목에 "JWT" fuzzy 매칭 |
| `/task C1` | `[C1]` 같은 리뷰 발견사항 ID 매칭 |
| `/task 로그인 테스트` | "로그인" + "테스트" 포함 task |
| `/task 기능: signup` | signup 기능 아래 다음 미완료 task |

후보가 여러 개면 사용자에게 선택지 제시.

### 1-2. 자동 선택 (인자 없을 때)

우선순위 순으로 탐색:
1. **진행 중 ⏳ 표시된 task** — 이전 작업 이어받기
2. **현재 Phase의 미완료 중 의존성 해결된 것**
3. 여러 개면 **상위 3개 후보** 제시 → 사용자 선택

---

## Phase 2: 맥락 자동 로드

선택된 task를 분석하여 관련 파일을 Read로 로드:

### 2-1. 기능 식별
- tasks.md에서 해당 task의 `### 기능: <X>` 헤더 찾기
- `### 공통 / 인프라` 아래면 공통으로 분류

### 2-2. 관련 파일 로드
기능이 식별되면:
- `docs/features/<X>/prd.md` — 요구사항 맥락
- `docs/features/<X>/tech-spec.md` — 기술 설계, 해당 task 관련 섹션

추가:
- `docs/plan.md` §6 (기술 스택)
- `CLAUDE.md` (프로젝트 원칙)
- `docs/context.md` — `[feature: <X>]` 태그 ADR 필터링하여 관련 결정 확인

### 2-3. 의존성 확인
task 설명에 "의존: X" 있으면 해당 선행 task 상태 확인. 미완료면 사용자에게 경고.

---

## Phase 3: 맥락 요약 제시

다음 형식으로 출력:

```markdown
📋 Next Task

**<task 제목>** [Effort]
- 소속: 기능: <X> > <카테고리 (구현/테스트/문서/리뷰 발견사항)>
- AC:
  - <AC 1>
  - <AC 2>
- 의존: <선행 task 제목 + 상태> (있으면)

**관련 설계**
- docs/features/<X>/tech-spec.md §<섹션 번호/제목>
- <핵심 설계 요약 1~2줄>

**관련 결정 (ADR)**
- <날짜> [feature: X] <결정 요약>
- <없으면 생략>

**시작하시겠어요?**
- "시작" / "진행" / Enter → 구현 대화 진입
- "상세" → 설계 문서의 해당 섹션 더 자세히
- "PRD 보기" → prd.md의 관련 부분 출력
- "다른 task" → 다른 task 선택지 제시
- "나중에" → 종료
```

---

## Phase 4: 구현 진입

사용자가 시작에 동의하면:

### 4-1. tasks.md 진행 중 표시
해당 task 옆에 `⏳` 또는 `(진행 중)` 표시 추가 (체크박스는 그대로 `[ ]`).

### 4-2. 자연 대화로 구현
- AC 기반으로 구현
- 사용자 요청에 따라 코드 작성 / 수정
- Critical 보안 이슈나 아키텍처 결정이 중간에 나오면 기억해둠 (Phase 5에서 update-docs 제안 근거)

---

## Phase 5: Task 완료 처리

사용자가 "완료", "됐다", "끝났어" 등으로 완료 표시할 때.

### 5-1. AC 검증 (완료 전 확인)

AC 항목별로 간단히 체크. 방법:

1. **코드 변경 확인**: `git diff` 로 관련 파일 변경 내용 살펴보기.
2. **테스트 실행** (있을 경우): Bash로 관련 단위/통합 테스트 실행. 실패 시 사용자에게 알림.
3. **수동 항목**: 코드로 확인 불가한 AC (예: "화면에 X 표시됨")는 사용자에게 한 번 확인:
   ```
   이 task의 AC 체크:
   - ✅ <자동 확인된 항목>
   - ❓ <수동 확인 필요 항목> — 확인하셨나요?
   ```

모든 AC가 충족됐다는 합의가 있어야 5-2 진행.
단, 사용자가 "AC 일부 미충족이지만 완료로 처리" 요청 시 메모 남기고 진행 (예: tasks.md 체크오프 옆에 "(AC X 미충족 — 추가 task 필요)" 주석).

### 5-2. tasks.md 체크오프 (원래 위치에)

사용자 동의 후:
1. 해당 task의 `[ ]` → `[x]`
2. `⏳ (진행 중)` 표시 있으면 제거
3. 완료 시각 추가: `[x] **<제목>** [M] ✅ <YYYY-MM-DD>`
4. **이동하지 않는다.** 원래 있던 `### 기능: <X> > #### 카테고리` 아래 그대로 유지.

> ❗ 이전 설계 변경: 개별 todo 완료 시 "완료된 작업" 섹션으로 이동했지만, 맥락 손실 문제로 **원래 위치 유지**로 변경됨. 기능 전체 완료 시에만 아카이브 (5-3 참고).

### 5-3. 기능 전체 완료 감지 + 아카이브 제안

체크오프 후, 해당 task가 속한 **기능 섹션** (`### 기능: <X>`) 의 **모든 하위 todo 가 [x] 인지 확인**:

- 카테고리(`#### 구현`, `#### 테스트`, `#### 문서화`, `#### 리뷰 발견사항`) 전부 순회
- 미완료 todo 1개라도 있으면 → 아카이브 제안 **하지 않음**
- 전부 완료됐으면 → 아래 제안:

```
🎉 기능 "<X>" 전체 완료 감지.
하위 <N개> todo가 모두 완료됐습니다.

"완료된 기능" 섹션으로 아카이브할까요?
- "네" / "Y" → 기능 섹션 전체를 tasks.md "✅ 완료된 기능" 아래로 이동
                  (기능명 옆에 완료일 추가)
- "나중에" → 그대로 유지 (진행 중 섹션에 남음)
```

**승인 시 동작**:
1. Phase 아래 `### 기능: <X>` 섹션 전체를 잘라냄
2. `## ✅ 완료된 기능` 아래로 붙여넣기
3. 섹션 제목에 완료일 추가: `### 기능: <X> (완료: <YYYY-MM-DD>)`
4. 하위 구조(구현/테스트/문서/리뷰 발견사항) 그대로 유지

### 5-4. AI 판단: 주요 변경 감지 (update-docs 제안)

이번 task에서 다음 중 하나가 있었으면 **`/update-docs` 를 권장**:
- 🔔 **주요 기술 결정** (예: 라이브러리 선택 변경, 아키텍처 패턴 결정)
- 🔔 **외부 API / 의존성 추가**
- 🔔 **DB 스키마 변경**
- 🔔 **context.md 제약사항에 영향**
- 🔔 **3개 이상 파일에 걸친 큰 변경**

그 외 일반 구현은 update-docs 권장 생략.

### 5-5. 연속성 제안 (후속 액션)

```markdown
✅ 완료: <task 제목>

다음을 진행하시겠어요?

**기본 옵션**
- 다음 task: `/task` 또는 "다음" → 자동으로 다음 ⏳ 항목 진입
- 일단 휴식: "나중에" → 종료

**상황별 옵션**
- (AI가 주요 변경 감지 시) 💡 주요 결정 있었음 (<요약>). `/update-docs` 권장.
- (기능 내 모든 task 완료 시) 🎉 기능 "<X>" 전체 완료. `/code-review` 로 검토 권장.
- (리뷰 발견사항 수정 완료 시) 🔧 리뷰 이슈 모두 처리. `/update-docs` 로 완료 반영 권장.
- (완전히 막혔으면) 📚 다른 task로 전환? 또는 `/project-status` 로 전체 상황 재확인.
```

### 5-6. 사용자 "다음" / "`/task`" 응답 시

→ Phase 1로 돌아가 다음 task 진입 루프. 맥락은 계속 유지.

---

## 핵심 원칙 (위반 금지)

1. **맥락 로드까지만**: 구현은 자연 대화. AI가 승인 없이 코드 안 씀.
2. **체크오프는 사용자 동의 후**: "완료"라고 명확히 밝힐 때만.
3. **완료 항목은 이동, 삭제 X**: "완료된 작업" 섹션에 스탬프와 함께 보존.
4. **의존성 존중**: 선행 task 미완료면 경고.
5. **update-docs 제안은 AI 판단 근거 명시**: "주요 결정 있었다" 대신 "라이브러리 Y → Z 변경됨 같은 구체 이유".
6. **매번 update-docs 강요 금지**: 경량 task는 tasks.md 체크오프로 충분.

## 실패 시나리오

| 상황 | 대응 |
|---|---|
| tasks.md 전부 완료 | `/feature-start` 또는 `/project-status` 안내 후 종료 |
| 인자 매칭 실패 | "해당 task를 찾을 수 없다. 후보 3개 제시" |
| 의존성 미해결 | 경고 후 진행 여부 묻기 (강제 차단 X — 사용자 판단) |
| prd/tech-spec 누락 | 경고 + "설계 문서 없이 진행하겠습니까?" 묻기 |
| 사용자가 task 아닌 잡담 시작 | 구현 진입 전 자연스럽게 응답. 단 "그럼 시작할까요?" 재확인 |
| 구현 중 범위 벗어남 (PRD 밖 기능 요청) | 경고 후 "기능 확장이 필요하면 `/feature-start` 또는 PRD 수정 권장" |
